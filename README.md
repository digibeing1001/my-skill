什么是 my-skill？
my-skill 是一个基于 OpenClaw 的技能创建工具。

但它不是那种给你模板、让你填参数的"文档助手"。它更像一个对话伙伴。

你跟它说"我想做一件什么事"，然后它会用苏格拉底式追问，一层层剥开你最初的表述，直到找到那个最根本的东西：

这个 Skill 到底要解决什么问题？
最不可省略的那一步是什么？
怎么才算做好了？
整个过程不需要你懂技术。不需要你会写代码。不需要你知道什么 YAML、什么 schema。

你只需要回答问题。

特性

🗣️ 对话式创建 — 像聊天一样，告诉 AI 你想做什么

🔍 第一性原理追问 — 层层剥开模糊需求，找到本质

🚫 零技术门槛 — 不需要写代码、懂 YAML、会配置

♻️ 自动迭代优化 — 同一问题累计 3 次自动沉淀改进

## 第一性原理三问框架

> 每个 Skill 的根基，由以下三个层次构成：

| 层次 | 核心问题 | 追问目标 | 不推进的条件 |
|---|---|---|---|
| **第一层** | 这个 Skill 究竟解决什么问题？ | 找到**根本目的**，剥离表面需求 | 用户没说清楚"谁在什么情况下面临什么问题" |
| **第二层** | 最不可简化的核心动作是什么？ | 界定**能力内核**，找到最关键一步 | 用户没说清楚"具体动作或判断"及"输入/输出" |
| **第三层** | 如何判断它成功了？ | 定义**成功标准**，让 AI 知道何时停止 | 用户没说清楚"具体要拿到什么"和"怎么验证" |

## 对话阶段流程

| 阶段 | 名称 | AI 第一句话 | 用户需回答 |
|---|---|---|---|
| **0** | 启动 | "你想做一件什么事？" | 你想做什么 |
| **1** | 锚定本质意图 | "你说的……根本原因是什么？" | 为什么需要这个 Skill |
| **2** | 界定能力内核 | "最难、最费时间、最容易出错的是哪一步？" | 哪一步最核心 |
| **3** | 定义输入与成功标准 | "Skill 必须知道的最少信息是什么？怎么判断做好了？" | 输入什么、输出什么、什么算成功 |
| **4** | 确认心智蓝图 | "我帮你总结一下……是这样吗？" | 确认/OK |
| **5** | 边界与极端情况 | "如果信息不全或给错了，Skill 怎么办？" | 出问题时的处理方式（可选） |
| **6** | 辅助需求 | "需要配合什么脚本、模板、参考资料吗？" | 辅助文件需求（可选） |
| **7** | 确认与生成 | "以后你说什么话，可以让 Skill 知道'我需要这个技能'？" | 触发词 |

## 追问规则（铁律）

| 规则 | 说明 |
|---|---|
| **一次只问一个** | 不问一串，只问最触及核心的那一个 |
| **基于回答生成** | 不套模板，根据用户说的内容决定下一个问题 |
| **不够具体不推进** | 模糊回答（"大概"、"可能"）必须追问到底 |
| **至少保持五轮追问** | 每轮必须有明确信息增量；简单技能五轮内完成；复杂技能可超过五轮 |
| **不得以"意图已清晰"为由跳过追问** | 五轮追问是下限，不是上限 |
| **大白话输出** | 所有问题让普通人看得懂，不出现技术术语 |

---

## 各阶段"够具体"的判断标准

### 第一层（锚定本质意图）

- ✅ 用户说了**谁**在什么**情况**下面临什么问题
- ✅ 用户说了**期望的结果**而不是实现方式
- ❌ 不含"大概"、"可能"、"就是为了……"

### 第二层（界定能力内核）

- ✅ 用户说了**具体动作或判断**（不是"整个过程"）
- ✅ 用户说了这一步的**输入和输出**
- ✅ 用户说了为什么这一步**不可替代**

### 第三层（定义输入与成功标准）

- ✅ 用户说了**具体要拿到什么东西**（不是"处理一下"）
- ✅ 用户说了**怎么判断成功**（具体可验证的标准）
- ✅ 用户说了**输入的形态**（文字/文件/链接/其他）

## 自我迭代优化机制

| 项目 | 说明 |
|---|---|
| **触发条件** | 同一类型的用户调整指令累计 ≥ 3 次 |
| **执行流程** | 用户提调整 → 记录日志 → 判断是否同类型 → 累计 ≥ 3 次 → 触发沉淀 |
| **沉淀输出** | AI 提出改进建议 → 用户确认 → 写入 Skill 规则 |

## 最终产出物

> 对话完成后，生成完整的 `SKILL.md` 文件，包含以下章节：

| 章节 | 内容 |
|---|---|
| 名称与定位 | 一句话说明这个 Skill 是什么 |
| 使用场景 | 什么时候用 / 什么时候别用 |
| 输入与输出 | 你给它什么，它给你返回什么 |
| 执行流程 | 每一步怎么走，产出什么 |
| 成功标准 | 怎么判断 Skill 做得好 |
| 边界处理 | 信息不全或给错了怎么办 |
| 触发词 | 以后你说什么话，AI 知道需要这个技能 |
| 第一性原理追问规则 | 内置追问轮次与原则 |
| 自我迭代机制 | 同一问题累计3次自动沉淀改进 |

## 独特优势

| 对比维度 | 传统方式 | my-skill |
|---|---|---|
| **起点** | 用户自己描述需求（往往模糊） | AI 通过追问帮你澄清真实需求 |
| **方法** | 给模板、让用户填参数 | 第一性原理三问，剥到本质 |
| **技术门槛** | 需要懂 YAML、目录结构、格式规范 | 零门槛，只需要回答问题 |
| **产出质量** | 取决于用户描述能力 | 模糊需求不被推进，直到够具体 |
| **迭代方式** | 手动修改文件 | 同一问题出现3次自动沉淀改进 |
| **价值主张** | 给你一个工具 | 做你的思维伙伴，帮你造工具 |
